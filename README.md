# railoptim

Сервис оптимизации назначений порожних вагонов под погрузку.

На вход поступают узлы образования порожних вагонов (предложение) и узлы погрузки
(спрос), тарифы, сроки доставки и набор бизнес-ограничений
(инфраструктура, промывка, ёмкости отстоя, станции массовой выгрузки и т.д.).
На выходе — назначения «откуда → куда», минимизирующие суммарную стоимость
плеч при полном покрытии спроса, насколько это возможно.

Ядро — трёхслойная оптимизация:

1. **Greedy** — быстрое допустимое начальное решение.
2. **MIP** (HiGHS branch-and-cut) — строгое решение с жёстким учётом партийности
   массовой выгрузки, с warm-start из greedy и лимитом 120 с.
3. **ALNS** (Adaptive Large Neighbourhood Search) — метаэвристика поверх MIP;
   **пропускается**, если MIP доказал глобальный оптимум (`gap ≈ 0`).

## Данные и доменные сущности

### Узлы

| Сущность       | Что описывает                                                              |
|----------------|----------------------------------------------------------------------------|
| `SupplyNode`   | Порожние вагоны в точке образования: тип, ЕТСНГ последнего груза, период поставки (1 или 10 сут.), флаг массовой выгрузки, статус ремонта, флаг `Assigned`. |
| `DemandNode`   | Потребность в вагонах: тип, код станции, срок, `purpose` — `Load` (погрузка) либо `Wash` (ёмкость промывки). |
| `TariffNode`   | Стоимость и срок доставки порожнего вагона между станциями.               |
| `TaskArc`      | Дуга графа `SupplyNode → DemandNode` с уже учтёнными фильтрами и стоимостью. |

### Разделение предложения перед оптимизацией

`SupplyNode` из API разделяются на три группы:

1. **Assigned** — уже назначены по факту; не участвуют в оптимизации.
2. **NeedsRepair** — требуют ремонта; идут в отдельный отчёт «В ремонт».
3. **opt_supply** — свободные вагоны; именно они попадают в оптимизацию.

Узлы спроса с `purpose = Wash` обрабатываются как «ёмкость промывки» — верхний
предел без штрафа за недобор; `Load` — жёсткое равенство + штраф за неудовлетворённый спрос.

### Допустимость дуги (`TaskArc`)

Дуга попадает в оптимизацию, если проходит все жёсткие фильтры в
`build_task_arcs`:

- есть тариф (иначе дуга отбрасывается),
- совместимость типа вагона (`car_type_ok`),
- срок доставки укладывается в окно погрузки (`period_ok`; мягкое нарушение
  допускается со штрафом — см. `PER_DAY_DELIVERY_PERIOD_VIOLATION_PENALTY_RUB`),
- грязный вагон (по ЕТСНГ последнего груза) не направляется напрямую в несовместимый
  груз — для таких случаев маршрут принудительно удлиняется через станцию промывки
  с надбавкой `WASH_PATH_SURCHARGE_RUB`.

Стоимость дуги = тариф + надбавки периода + надбавки промывки + надбавка period 10
(`PERIOD10_COST_SURCHARGE_RUB`).

## Оптимизационное ядро

### 1. Greedy — начальное допустимое решение (`solver/greedy.rs`)

Сортирует допустимые дуги по `(cost, distance)` и жадно закрывает спрос,
уменьшая `remaining_supply`/`remaining_demand`. Ограничение **MIN_BATCH на
станциях массовой выгрузки** проверяется inline по двум условиям:

- **(A) пара не наберёт MIN_BATCH суммарно**: если
  `existing + station_remaining < MIN_BATCH` — дуга пропускается;
- **(B) застрявший остаток**: если назначение оставит на узле остаток
  `0 < residual < MIN_BATCH` и на других узлах той же станции тоже нечего
  добрать — дуга пропускается, чтобы не «замуровать» вагоны.

Результат — быстрое, честное покрытие; используется как warm-start для MIP и
как seed ALNS (либо прямой fallback, если MIP не запустился).

### 2. LP (`solver/lp.rs`)

Классическая транспортная LP (HiGHS simplex) с dummy-узлами:

- `dummy_demand` поглощает избыток предложения (стоимость 0);
- `dummy_supply` покрывает неудовлетворённый спрос штрафом
  `PENALTY_COST = 400 000 ₽/ваг.`, делая модель всегда допустимой.

LP **не умеет** выражать «0 или ≥ MIN_BATCH» — поэтому используется только как
быстрый репейр в ALNS (с inline-пост-проверкой) и как эталон для валидации.

### 3. MIP (`solver/mip.rs`)

MIP-постановка транспортной задачи с жёстким ограничением MIN_BATCH через
**big-M формулировку** (аналог `example.py`, строки 174 и 232–244).

Для каждой пары `(mass_supply_station, demand_station)` вводится бинарная
переменная `y_pair ∈ {0, 1}` и две строки:

```text
B_pair * y_pair  ≤  Σ x[arc]           (Σ = 0 либо Σ ≥ B_pair при y = 1)
Σ x[arc]         ≤  M_pair * y_pair    (поток по паре возможен только при y = 1)
```

где `B_pair = min(MIN_BATCH_FROM_MASS_STATION, station_supply)`,
`M_pair = station_supply` — тонкая верхняя оценка, улучшающая LP-релаксацию.

Все дуговые переменные целочисленные (`add_integer_column`); dummy-узлы —
непрерывные (их целочисленность автоматически обеспечивается целочисленностью
дуг и целыми правыми частями).

**Warm-start** из greedy: значения дуг берутся из жадного решения,
dummy-переменные и `y_pair` достраиваются по соответствующим балансам.
Это даёт HiGHS готовый incumbent и сразу отсекает значительную часть дерева
поиска.

**Параметры решателя**:
- `time_limit` — 120 с по умолчанию (`DEFAULT_MIP_TIME_LIMIT`);
- `mip_rel_gap` — 0.5% по умолчанию (`DEFAULT_MIP_REL_GAP`);
- `presolve = on`, `parallel = on`, 8 потоков.

Возвращает [`MipOutcome`](src/solver/mip.rs) со статусом HiGHS, MIP-gap и
значениями дуговых переменных. Два ключевых предиката:

- `is_globally_optimal()` — `Optimal && gap < 1e-6`: HiGHS **доказал**
  оптимальность. ALNS в этом случае пропускается.
- `has_feasible_solution()` — есть ли хоть какой-то incumbent (для fallback
  в `repair_mip`).

### 4. ALNS (`solver/alns.rs`)

Adaptive Large Neighbourhood Search — метаэвристика вокруг destroy/repair.

Запускается **только если MIP не нашёл глобальный оптимум**; стартует с
лучшего из `greedy` и `MIP-incumbent` по лексикографическому сравнению
`(покрытие, −стоимость)`.

**Цикл одной итерации**:

1. `destroy_random` — удаление `k = K·|state|` случайных назначений с
   возвратом вагонов в остатки.
2. `drain_violated_mass_pairs` — после destroy суммарный поток по паре
   массовой выгрузки мог упасть ниже MIN_BATCH; такие пары эвакуируются
   целиком, их спрос возвращается в очередь ремонта.
3. **Repair** — один из операторов:
   - `repair_mip` (по умолчанию) — MIP на подзадаче из дуг вокруг разрушенных
     узлов с тонкими лимитами (`ALNS_MIP_TIME_LIMIT = 3 с`,
     `ALNS_MIP_REL_GAP = 2%`). Жёсткий MIN_BATCH встроен в модель, поэтому
     назначения не «теряются» на этапе применения.
   - `repair_lp` — быстрый LP-ремонт с inline-пост-проверкой MIN_BATCH
     (оставлен для сравнения и как fallback по флагу `use_mip_repair = false`).
   - `repair_greedy` — жадный fallback, если ни MIP, ни LP не вернули
     пригодного решения.
4. **Accept** — лексикографический критерий:
   - сначала минимизируем `unmet + excess` (покрытие),
   - при равенстве — минимизируем реальную стоимость `total_cost`.

   Такое сравнение надёжнее, чем агрегированная `objective_cost`:
   при `PENALTY_COST = 400 000 ₽/ваг.` один «потерянный» вагон перекрывает
   сотни тысяч экономии, но сравнение дробных `f64` на границе ненадёжно.
5. **Adapt K** — доля разрушения (`destroy_ratio`) плавно растёт при стагнации
   (`STAGNATION_THRESHOLD = 50` итераций без улучшения) и уменьшается при
   каждом улучшении. Диапазон `[0.05, 0.50]`.

Бюджет времени — `DEFAULT_TIME_BUDGET = 180 с`. Результат —
[`AlnsResult`](src/solver/alns.rs): лучшее состояние, значения дуговых
переменных и статистика итераций.

## Запуск

```bash
# разовый запуск с секретами из Infisical (пароли БД, API-токены)
./run.sh

# или напрямую после экспорта переменных окружения
cargo run --release
```

Секреты извлекаются только из self-hosted Infisical
(см. `auth-infisical.sh`); в коде и репозитории их быть не должно.

## Технологический стек

- **Язык**: Rust 2024, async runtime — `tokio`.
- **Solver**: [HiGHS](https://highs.dev/) 2.0 — LP (simplex) и MIP (branch-and-cut).
- **Источники данных**: REST API (через `reqwest` + `serde`) и MSSQL.
- **Ошибки**: `anyhow` на верхнем уровне, `thiserror` в модулях.
- **Выходные данные**: JSON-отчёт и xlsx (через `rust_xlsxwriter`); финальные
  назначения отправляются обратно в API.

---

## Конвенции кода

1. **Error handling**: `anyhow::Result` в `main`, кастомные enum-ошибки с
   `thiserror` в модулях.
2. **Типизация**: строгие `serde`-структуры для всех внешних JSON; никаких
   `serde_json::Value` без крайней необходимости.
3. **Async**: функции в `data/`, взаимодействующие с API/БД, — `async`.
   Ядро солвера — синхронное: HiGHS требует монопольного владения моделью.
4. **Безопасность**: секреты только из Infisical, никаких хардкод-токенов,
   никаких `.env` в git.
